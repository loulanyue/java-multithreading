# Deadlock Troubleshooting

死锁是 Java 多线程面试里最经典的追问之一，也是线上问题里最让人头疼的一类并发故障。这份文档聚焦“怎么理解、怎么发现、怎么排查”。

## 什么是死锁

死锁可以简单理解成：多个线程互相等待对方释放资源，结果谁也无法继续执行。

## 四个经典条件

通常会提到以下四个条件：

1. 互斥：资源同一时刻只能被一个线程占用
2. 占有并等待：线程已经拿到一部分资源，又继续等待其他资源
3. 不可剥夺：资源不能被强制抢走
4. 循环等待：线程之间形成等待环

## 面试里的典型例子

最常见的解释方式是两个线程、两把锁：

- 线程 A 先拿锁 1，再等锁 2
- 线程 B 先拿锁 2，再等锁 1

如果两个线程都已经拿到第一把锁，就可能互相卡住。

## 如何排查

可以按这个思路回答：

1. 先看程序是否卡死、线程是否长期无响应
2. 用 `jstack` 导出线程栈
3. 检查是否存在线程互相等待对方持有的锁
4. 结合业务代码定位锁获取顺序

## `jstack` 关注点

面试里常见回答：

- 通过 `jstack` 看线程状态
- 查找 `Found one Java-level deadlock` 之类的提示
- 对照堆栈里的锁信息找出具体类和代码位置

## 如何避免

- 保持一致的加锁顺序
- 尽量缩小锁粒度
- 避免在持有锁时执行耗时操作
- 能用并发容器或更高层同步工具时，尽量不要手写复杂锁嵌套

## 面试回答建议

回答死锁时，最好按这个顺序组织：

1. 先定义死锁
2. 再讲经典成因
3. 再补排查方法
4. 最后说预防思路

这样比只背“四个条件”更完整，也更像真正做过排查的人。
